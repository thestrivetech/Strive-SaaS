5. DATA FLOW & ARCHITECTURE PATTERNS
================================================================================

5.1 REQUEST FLOW (Server Components)
-------------------------------------
1. User navigates to page (e.g., /platform/crm)
2. Next.js renders Server Component (app/app/(platform)/crm/page.tsx)
3. Server Component calls data function:
   - import { getCustomers } from '@/lib/modules/crm/queries'
   - const customers = await getCustomers()
4. Query function uses Prisma:
   - const customers = await prisma.customer.findMany({
       where: { organizationId }
     })
5. Supabase RLS automatically filters by organization
6. Data returned, component renders
7. HTML streamed to client
8. Client hydrates interactive components only

**No API route needed! Direct database access in Server Components.**


5.2 MUTATION FLOW (Server Actions)
-----------------------------------
1. User interacts with form (Client Component)
2. Form calls Server Action:
   - import { createCustomer } from '@/lib/modules/crm/actions'
   - const result = await createCustomer(formData)
3. Server Action validates with Zod:
   - const validated = CustomerSchema.parse(data)
4. Server Action performs database operation:
   - const customer = await prisma.customer.create({ data: validated })
5. Server Action revalidates Next.js cache:
   - revalidatePath('/platform/crm')
6. Result returned to client
7. UI automatically updates with fresh data

**Progressive Enhancement: Works without JavaScript!**


5.3 REALTIME UPDATES FLOW
--------------------------
1. Component subscribes to Supabase Realtime channel
2. Database change occurs (INSERT, UPDATE, DELETE)
3. Supabase Realtime broadcasts event to subscribers
4. Component receives event and updates UI
5. Uses optimistic updates for instant feedback

Example: Task board automatically updates when another user creates a task


5.4 AUTHENTICATION FLOW
------------------------
1. User submits login form
2. Server Action calls Supabase Auth:
   - const { data } = await supabase.auth.signInWithPassword({
       email, password
     })
3. Supabase returns JWT token
4. Token stored in httpOnly cookie (secure)
5. Subsequent requests include cookie automatically
6. Middleware extracts user from JWT:
   - const { data: { user } } = await supabase.auth.getUser()
7. Middleware checks organization membership
8. Request proceeds if authorized, redirects if not


5.5 RBAC (Role-Based Access Control) FLOW
------------------------------------------
1. Middleware extracts user from session
2. Looks up organization membership:
   - const member = await prisma.organizationMember.findUnique({
       where: { userId_organizationId }
     })
3. Checks user role: owner | admin | manager | member | guest
4. Compares against required permission for route
5. Allows or denies access
6. In components, use: `const { hasPermission } = usePermissions()`


5.6 FILE UPLOAD FLOW
--------------------
1. User selects file in Client Component
2. File uploaded to Supabase Storage:
   - const { data } = await supabase.storage
       .from('bucket')
       .upload('path', file)
3. URL returned: https://[project].supabase.co/storage/v1/object/public/...
4. URL saved to database via Server Action
5. File served with RLS policies enforcing access control


5.7 AI CHAT FLOW
----------------
1. User sends message (Client Component)
2. Server Action receives message
3. Retrieves conversation history from database
4. Optionally performs RAG search:
   - Query vector database for relevant context
   - Use pgvector similarity search in Supabase
5. Constructs prompt with context
6. Calls AI provider (OpenRouter/Groq):
   - Streams response back to client
7. Saves message and response to database
8. Returns streaming response to UI

6. KEY ARCHITECTURAL DECISIONS
================================================================================

6.1 WHY PRISMA + SUPABASE (NOT EITHER/OR)
------------------------------------------
**Decision:** Use both Prisma and Supabase together

**Rationale:**
â€¢ Supabase provides:
  - Database hosting (PostgreSQL)
  - Authentication (pre-built, secure)
  - Storage (file management)
  - Realtime (live updates)
  - RLS (data isolation)

â€¢ Prisma provides:
  - Type-safe database queries
  - Complex aggregations and transactions
  - Schema migrations
  - Better TypeScript integration
  - Easier testing (mock Prisma client)

**Pattern:**
  All database queries go through Prisma â†’ Supabase PostgreSQL
  Auth, Storage, Realtime go through Supabase SDK directly

**Files:**
  - lib/supabase/client.ts - Supabase client-side SDK
  - lib/supabase/server.ts - Supabase server-side SDK
  - lib/prisma.ts - Prisma client singleton


6.2 WHY SERVER-FIRST (NOT CLIENT-HEAVY SPA)
--------------------------------------------
**Decision:** Default to Server Components, minimize client JavaScript

**Rationale:**
â€¢ Better initial page load (no JS needed for static content)
â€¢ Improved SEO (fully rendered HTML)
â€¢ Reduced bundle size (code runs on server)
â€¢ Direct database access (no API layer needed)
â€¢ Better security (credentials never exposed)
â€¢ Automatic code splitting
â€¢ Progressive enhancement (works without JS)

**Trade-off:** Slightly more complex mental model, but much better UX


6.3 WHY MODULE ISOLATION (NO CROSS-MODULE IMPORTS)
---------------------------------------------------
**Decision:** Modules cannot import from other modules

**Rationale:**
â€¢ Prevents circular dependencies
â€¢ Enforces clear boundaries
â€¢ Makes modules easier to understand
â€¢ Allows independent development
â€¢ Facilitates future extraction (if needed)

**Data Sharing:**
  Use Prisma-generated types from @prisma/client
  Example: import type { Customer } from '@prisma/client'


6.4 WHY 500-LINE FILE LIMIT
----------------------------
**Decision:** Hard limit of 500 lines per file (enforced by ESLint)

**Rationale:**
â€¢ Forces decomposition and modularity
â€¢ Improves readability
â€¢ Makes testing easier
â€¢ Encourages reusable components
â€¢ Prevents "God objects"

**Targets:**
  - UI Components: 200 lines
  - Server Components: 250 lines
  - Services/Logic: 300 lines
  - API Routes: 150 lines

**Enforcement:** ESLint blocks commits if exceeded


6.5 WHY TDD (TEST-DRIVEN DEVELOPMENT)
--------------------------------------
**Decision:** Write tests BEFORE implementation

**Rationale:**
â€¢ Better API design (think about usage first)
â€¢ Higher code coverage (80% minimum enforced)
â€¢ Fewer bugs in production
â€¢ Living documentation
â€¢ Confidence when refactoring
â€¢ Faster debugging

**Requirement:** All Server Actions and API routes MUST have tests

7. DEVELOPMENT WORKFLOW
================================================================================

7.1 READ-BEFORE-EDIT MANDATE
-----------------------------
**CRITICAL: Follow these steps BEFORE any code change:**

1. **READ FIRST** - Use Read tool on any file before editing
2. **SEARCH FOR EXISTING** - Use Glob/Grep to check if similar code exists
   - Don't create duplicate test files
   - Don't recreate existing utilities
   - Don't duplicate components
3. **UPDATE, DON'T CREATE** - Prefer editing over creating (99% of time)
4. **ASK IF UNCERTAIN** - When unsure, ask before proceeding

**Rationale:** Prevents code duplication, maintains consistency


7.2 FEATURE DEVELOPMENT WORKFLOW
---------------------------------
1. **Write tests first** (TDD)
   - Create test file: feature.test.ts
   - Write failing tests
   - Run: npm test

2. **Implement feature**
   - Create Server Action if mutation needed
   - Create query function if data fetching needed
   - Create Zod schema for validation
   - Create UI component

3. **Make tests pass**
   - Run: npm test -- --watch
   - Iterate until green

4. **Verify quality**
   - Run: npm run lint
   - Run: npx tsc --noEmit
   - Check coverage: npm test -- --coverage
   - Must be 80%+ or blocked

5. **Commit**
   - All checks must pass before commit
   - Commit message: Conventional Commits format


7.3 DATABASE CHANGES WORKFLOW
------------------------------
1. **Update Prisma schema**
   - Edit: prisma/schema.prisma
   - Add/modify models

2. **Create migration**
   - Run: npx prisma migrate dev --name description
   - Migration file created in prisma/migrations/

3. **Generate Prisma client**
   - Run: npx prisma generate
   - Types automatically updated

4. **Update code to use new schema**
   - TypeScript will show errors for old usage
   - Update queries to match new schema

5. **Update tests**
   - Update fixtures
   - Update test expectations


7.4 COMPONENT CREATION WORKFLOW
--------------------------------
1. **Check if similar component exists**
   - Search: components/ directory
   - Don't duplicate

2. **Choose component type**
   - UI primitive: components/ui/
   - Feature-specific: components/features/[module]/
   - Shared: components/shared/

3. **Determine if Server or Client Component**
   - Need interactivity (clicks, state)? â†’ Client ("use client")
   - Just rendering? â†’ Server (default, no directive)

4. **Create component file**
   - Use TypeScript
   - Add prop types interface
   - Export default or named export

5. **Test component**
   - Create component.test.tsx
   - Test user interactions
   - Test edge cases
